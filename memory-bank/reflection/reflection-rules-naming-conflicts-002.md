# Рефлексия: rules-naming-conflicts-002

## Summary

Кодовая база приведена в соответствие с `docs/common/naming-conventions.md`: интерфейсы переименованы в i*camelCase (`i_todo`, `i_todoState`, `i*\*Props`), обычные файлы — в kebab-case (`use-filter`, `use-todos`, `use-local-storage`, `todo-helpers`). Все импорты и использования типов обновлены. Тесты и сборка проходят.

## What Went Well

- Чёткий план по фазам (интерфейсы → файлы) и чеклист в tasks.md позволили выполнить переименования без пропусков.
- Использование `git mv` при переименовании файлов сохранило историю.
- Проверка конфигов (Vite, Vitest, TS) на явные пути к файлам — явных путей не было, регрессий не внесено.

## Challenges

- Много файлов и импортов; риск пропустить использование типа или пути. Митигация: поиск по старым именам перед коммитом.
- В среде EPERM при запуске Vitest/Vite — проверка тестов и сборки выполнена локально.

## Lessons Learned

- Детальный план и разбиение на фазы снижают риск при массовых переименованиях.
- Для переименования файлов полезно явно проверять конфиги на жёстко заданные пути.
